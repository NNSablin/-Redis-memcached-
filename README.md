### Задание 1. Кеширование
Приведите примеры проблем, которые может решить кеширование.

Приведите ответ в свободной форме.
1. Проблема: Высокая задержка (Latency)
Суть проблемы: Получение данных из медленного источника (например, базы данных на жестком диске, удаленного API или внешнего сервиса) занимает много времени.

Как решает кеширование: Данные сохраняются в быстродействующем хранилище (обычно в оперативной памяти), расположенном близко к потребителю. Последующие запросы получают данные из кеша за доли миллисекунд, а не сотни миллисекунд.

Пример: Загрузка профиля пользователя в социальной сети. Вместо каждого раза читать его из БД, данные кешируются в Redis/Memcached.

2. Проблема: Низкая пропускная способность (Throughput)
Суть проблемы: Система не может обработать большое количество запросов в единицу времени, так как "бутылочное горлышко" (например, база данных) перегружается.

Как решает кеширование: Кеш берет на себя значительную часть нагрузки (читающих запросов), снижая давление на основную систему. Это позволяет одному и тому же оборудованию обслуживать больше пользователей.

Пример: Черная пятница в интернет-магазине. Каталог товаров, который тысячи раз запрашивают пользователи, отдается из кеша, а не из перегруженной базы данных.

3. Проблема: Повторяющиеся тяжелые вычисления
Суть проблемы: Один и тот же сложный расчет выполняется многократно с одними и теми же входными данными, потребляя ценные ресурсы CPU.

Как решает кеширование: Результат вычисления сохраняется. При следующем запросе с теми же параметрами система просто возвращает готовый результат из кеша.

Пример: Генерация отчета "Продажи за прошлый месяц" или рендеринг сложной шапки сайта. Вместо того чтобы строить отчет заново для каждого менеджера, результат кешируется на час.

4. Проблема: Низкая отказоустойчивость и доступность
Суть проблемы: При сбое основного источника данных (БД, внешний API) приложение полностью перестает работать.

Как решает кеширование: Кеш может какое-то время отдавать устаревшие данные (стратегия "Cache-Aside"), позволяя основному функционалу приложения работать, пока база данных восстанавливается. Это повышает устойчивость системы к сбоям.

Пример: Интернет-магазин продолжает показывать каталог товаров (пусть и с возможным устаревшим остатком на складе), даже если база данных временно недоступна.

5. Проблема: "Горячие" данные (Hot Data)
Суть проблемы: Небольшой процент данных запрашивается несоразмерно часто (например, лента новостей знаменитости, информация о суперпопулярном товаре).

Как решает кеширование: Эти "горячие" данные помещаются в кеш, где к ним обеспечивается мгновенный доступ, предотвращая "точечную" перегрузку базы данных.

6. Проблема: Согласованность при высокой нагрузке (в определенных сценариях)
Суть проблемы: В распределенных системах обеспечить строгую согласованность данных между всеми узлами сложно и дорого.

Как решает кеширование: Используется для реализации ** eventual consistency** ( eventual согласованности). Данные кешируются на определенное время (TTL), и все узлы в течение этого периода видят одну и ту же версию данных, даже если где-то они уже изменились.

7. Проблема: Стоимость операций
Суть проблемы: Каждый запрос к внешнему платному API или дорогостоящему сервису (например, машинное обучение, сложные гео-расчеты) стоит денег.

Как решает кеширование: Кеширование результатов таких запросов позволяет значительно сократить их количество и, следовательно, затраты.

Пример: Кеширование ответов от геокодирующего сервиса, который преобразует адрес в координаты. Один раз преобразовали адрес "Красная площадь, 1" и закешировали результат на долгий срок.

Резюме: Кеширование — это не просто "ускорение". Это стратегический механизм, который решает фундаментальные проблемы современных распределенных систем: задержку, масштабируемость, устойчивость к сбоям и стоимость, перенося нагрузку с медленных и дорогих ресурсов на быстрые и дешевые.
